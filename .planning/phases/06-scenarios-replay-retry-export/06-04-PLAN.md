  <name>Task 1: ReplayEngine — pure logic (scrubTo + play/pause/speed)</name>
  <files>src/replay/ReplayEngine.js, tests/replayEngine.test.js</files>
  <behavior>
    Konstruktor:
    ```
    new ReplayEngine({ liveStore, gsapTicker })
    ```
    `liveStore` to istniejący Application store (do mutacji widoku 3D); `gsapTicker` to `gsap.ticker` obiekt (testy DI mockują z `{add(cb), remove(cb)}`).

    Stan instancji: `_events`, `_scenario`, `_eventIdx`, `_cursor` (ms od startu), `_paused: true`, `_speed: 1.0`, `_startTimestamp`, `_tickerCallback`.

    API:
    - `loadAttempt(attempt, scenario)` — przyjmuje `attempt` z session.attempts[N] (kształt `{events, scoring}`) i scenario object. Resetuje cursor=0, eventIdx=0, paused=true. `_startTimestamp = events[0]?.timestamp ?? 0`.
    - `play()` — `_paused = false`. Idempotent.
    - `pause()` — `_paused = true`. Idempotent.
    - `setSpeed(speed)` — `_speed = speed` (1.0 lub 0.25). Walidacja: throw gdy !== 1.0 && !== 0.25.
    - `scrubTo(targetIdx)` — re-execution z fresh store: `const fresh = createTrainingStore(); fresh.getState().startScenario(this._scenario); for (i=0; i<=targetIdx; i++) this._applyEventToStore(fresh, this._events[i])`. Po pętli — kopiuje `fresh.getState()` slice'y (`steps, currentStepId, machineState, meshStates, scoring, _currentAngle`) i `set` na `this._liveStore`. Aktualizuje `_eventIdx = targetIdx + 1`, `_cursor = (events[targetIdx].timestamp - _startTimestamp) ?? 0`.
    - `getCurrentPosition()` — zwraca `{eventIdx, cursor, totalEvents, totalDurationMs}` (totalDurationMs = last.timestamp - first.timestamp).
    - `attach()` — `this._tickerCallback = (_, dt) => this._onTick(dt); this._gsapTicker.add(this._tickerCallback)`.
    - `detach()` — `this._gsapTicker.remove(this._tickerCallback)`.
    - `dispose()` — `detach()` + reset state.
    - `onPositionChange(callback)` — rejestruje listener wołany po każdym applied event. Dla UI update.

    `_applyEventToStore(store, event)` — pure dispatcher:
    - `session.start` → no-op (już ustawione przez startScenario).
    - `step.done` → store.setState advance step + setCurrentAngle(event.angle ?? 0).
    - `step.violation` → store.setState ustawia stepStatus 'error' + setCurrentAngle.
    - `fault.triggered` → store.setState machineState = na podstawie faultId.
    - `session.spinUp.done` → store.setState machineState='gotowa-do-pracy'.

    UPROSZCZENIE: ReplayEngine NIE rekonstruuje WSZYSTKICH efektów (np. meshStates dla manipulacji) — to wymaga znajomości scenariusza. Zamiast: dla każdego `step.done` event'u ReplayEngine sięga do `this._scenario.steps.find(s => s.id === event.stepId)` i aplikuje step.effectsOnSuccess do storu. Dla `step.violation` — analogicznie effectsOnError. Ten approach == "deklaratywna re-execution".

    `_onTick(dt)`:
    ```
    if (_paused) return;
    _cursor += dt * _speed;
    while (_eventIdx < _events.length) {
      const ev = _events[_eventIdx];
      const evOffset = ev.timestamp - _startTimestamp;
      if (_cursor < evOffset) break;
      _applyEventToStore(_liveStore, ev);
      _eventIdx++;
      _onPositionChangeListeners.forEach(cb => cb(this.getCurrentPosition()));
    }
    if (_eventIdx >= _events.length) _paused = true;  // auto-pauza na końcu
    ```

    Boundary: ReplayEngine importuje TYLKO `createTrainingStore` (relative path do src/state/) + nic więcej. NIE THREE, NIE gsap (DI), NIE DOM, NIE @floating-ui. Zgłoś do boundaries.test.js (Task 3).

    Testy (`tests/replayEngine.test.js`):
    - new ReplayEngine + loadAttempt — instancja gotowa, paused=true, cursor=0.
    - play() → paused=false; pause() → paused=true.
    - setSpeed(0.25) → _speed=0.25; setSpeed(2.0) → throw.
    - scrubTo(0) z eventami [session.start, step.done s1] → store snapshot ma steps[s1].status==='done', currentStepId=następny.
    - scrubTo(0) potem scrubTo(1) potem scrubTo(0) → deterministic: state po drugim scrubTo(0) === state po pierwszym scrubTo(0) (porównaj steps + machineState).
    - _onTick z dt=1000ms i events.length=3 odpaloną od cursor=0 — odpala events które mają timestamp ≤ 1000.
    - _onTick speed=0.25 i dt=400ms → cursor wzrasta o 100ms (nie 400).
    - auto-pauza po przeskoczeniu wszystkich eventów.
    - attach/detach woła gsapTicker.add/remove.

    ~12 asercji.
  </behavior>
