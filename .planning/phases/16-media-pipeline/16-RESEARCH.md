## Summary

Phase 16 adds a media serving pipeline to the PM-300 Trener app. The core work is three components acting in concert: (1) `src/media/MediaManager.js` — a boundary-clean module that maps filenames to `/media/<filename>` absolute URLs (never JS imports) and validates URL reachability via injected fetch; (2) an extension to `ElementInfoOverlay._render()` that renders `<img>` elements for `entry.media[]` entries with `onerror` graceful fallback; (3) `public/media/ATTRIBUTION.txt` as a hard gate documenting license provenance of every file in `public/media/`.

The stack is already in place. The overlay's media slot (`<div class="element-info-overlay__media">`) exists at line 75 of `ElementInfoOverlay.js`; the render branch for when `entry.media` is empty exists at lines 258-265. Phase 16 extends that branch to render real `<img>` elements. The DI pattern for injecting a service into the overlay is already used for `lectorService`. The `fetch`-as-DI testability pattern is already proven by `LectorService` and its tests.

The biggest technical risk is `validateSrc()` testability in jsdom. The correct approach — confirmed by the existing codebase pattern — is to inject `fetchImpl` (or a default of `globalThis.fetch`) into `MediaManager`'s constructor, mirroring `LectorService`. This makes the method trivially mockable in tests without real network. The `new Image()` onload/onerror approach must NOT be used: jsdom 29 does not fire `onload`/`onerror` for `Image` objects set to non-existent URLs without a real HTTP server, making those events untestable.

**Primary recommendation:** Implement `MediaManager` with injected `fetchImpl` DI; add MediaManager as optional DI to `ElementInfoOverlay`; use `img.onerror` in the DOM (not MediaManager) for graceful UI degradation; populate 2-3 placeholder entries in `elementInfo.js`; write ATTRIBUTION.txt with placeholder provenance. Bundle headroom is 24 KB — placeholder webp files are tiny (< 5 KB each) and live in `public/`, so they never touch the JS bundle.

### Pattern 1: MediaManager with fetch DI (proven by LectorService)

```javascript
// src/media/MediaManager.js
// Boundary: TYLKO fetch (DI) + URL string. NIE three/gsap/state/ui/training.

export class MediaManager {
  /**
   * @param {{ fetchImpl?: Function }} [deps]
   */
  constructor({ fetchImpl } = {}) {
    this._fetch = fetchImpl ?? globalThis.fetch.bind(globalThis);
  }

  /** Zwraca absolutny URL publiczny (bez bundlowania przez Vite). */
  resolveSrc(filename) {
    return '/media/' + filename;
  }

  /** Sprawdza dosepnosc zasobu przez HEAD request. Zwraca Promise<boolean>. */
  validateSrc(src) {
    return this._fetch(src, { method: 'HEAD' })
      .then(r => r.ok)
      .catch(() => false);
  }
}
```

**Why `fetch HEAD` over `new Image()` onload/onerror:**
- `new Image()` onload/onerror do NOT fire in jsdom 29 without a real HTTP server. Tests would be untestable or require heavy setup.
- `fetch` via `fetchImpl` DI is already the proven pattern in this codebase (LectorService). Tests mock `fetchImpl = vi.fn().mockResolvedValue({ ok: true })`.
- `fetch HEAD` is correct for checking URL reachability without downloading the body. [ASSUMED — based on established web standards; no Context7 verification done for this fundamental fact]

### Pattern 2: ElementInfoOverlay media render with onerror degradation

```javascript
// Inside ElementInfoOverlay._render(), replacing the current media slot block:
// Lines 258-265 in current ElementInfoOverlay.js

if (mediaEl) {
  mediaEl.textContent = '';  // clear previous render
  if (!entry?.media?.length) {
    // No media — keep existing placeholder text behavior (Test 2d must pass).
    mediaEl.textContent = pl.modals.elementInfo.mediaPlaceholder;
  } else {
    entry.media.forEach((item) => {
      const src = this._mediaManager
        ? this._mediaManager.resolveSrc(item.src)
        : '/media/' + item.src;  // fallback if no MediaManager (backward-compat)
      const img = document.createElement('img');
      img.src = src;
      img.alt = item.alt ?? '';
      img.loading = 'lazy';
      img.className = 'element-info-overlay__media-img';
      // Graceful degradation: 404 / no network -> remove broken img silently.
      img.onerror = () => { img.remove(); };
      mediaEl.appendChild(img);
      if (item.caption) {
        const cap = document.createElement('p');
        cap.className = 'element-info-overlay__media-caption';
        cap.textContent = item.caption;
        mediaEl.appendChild(cap);
      }
    });
  }
}
```

**Critical:** `img.onerror` must be set BEFORE `img.src` in some browsers. Set it immediately after `createElement` — the pattern above is safe (both set before DOM append, but onerror is attached before src is set to allow immediate error if browser resolves synchronously).

**No `console.error` in onerror:** `img.onerror` silently removes the broken element. This satisfies the "ZERO JS console error" requirement in MED-03.

### Anti-Patterns to Avoid

- **`import img from './image.webp'`**: Vite would bundle the asset. Use `/media/<filename>` URL strings via `MediaManager.resolveSrc()`. Never use static JS asset imports for media.
- **`new Image()` onload/onerror for validateSrc()**: jsdom does not fire these events without a real server. Use `fetch HEAD` with DI instead.
- **Polish string literals in `src/media/MediaManager.js`**: The Polish literal scanner in `boundaries.test.js` would fail. MediaManager must have zero Polish string literals (it has no user-facing strings anyway — it's a pure service).
- **`img.onerror = console.error`**: Would violate the "ZERO JS console error" requirement of MED-03. Use silent removal instead.
- **Not clearing `mediaEl` before re-render**: `_render()` is called on every store change. Without `mediaEl.textContent = ''` at the start of the media block, images accumulate across re-renders.

## Common Pitfalls

### Pitfall 1: `new Image()` onload/onerror is untestable in jsdom
**What goes wrong:** `validateSrc()` implemented with `new Image(); img.onload = resolve; img.onerror = reject` — tests cannot trigger these callbacks without a real HTTP server. Tests hang or always return one outcome.
**Why it happens:** jsdom 29 does not simulate image loading. It stubs the `Image` constructor but does not fire load/error events for `src` assignments.
**How to avoid:** Use `fetch HEAD` with `fetchImpl` DI (see Pattern 1). Tests mock `fetchImpl = vi.fn().mockResolvedValue({ ok: false })`.
**Warning signs:** Test for `validateSrc` with a 404 URL always resolves `true` or never resolves.

### Pitfall 2: Assets referenced via JS imports get bundled
**What goes wrong:** A developer writes `import img from '../assets/photo.webp'` → Vite processes it → asset gets base64-inlined (or hashed into `dist/assets/`) → bundle grows → MED-01 violated.
**Why it happens:** Vite's default behavior is to process `import` statements pointing to asset files.
**How to avoid:** Never use JS asset imports for media. `MediaManager.resolveSrc()` returns a plain string `/media/filename.webp`. No import needed.
**Warning signs:** Build output shows `.webp` files in `dist/assets/` with hashed names, or bundle size jumps unexpectedly.

### Pitfall 3: `mediaEl` not cleared between renders causes duplicate images
**What goes wrong:** Store changes (e.g., mode switch, modal reopen) trigger `_render()`. If `mediaEl` is not cleared, each render appends new `<img>` elements on top of existing ones.
**Why it happens:** `_render()` is called on every subscription notification (activeModal, _elementInfoMeshId, mode, lectorEnabled, lectorVoiceId — 5 subscriptions).
**How to avoid:** Start the media block with `mediaEl.textContent = ''` or `mediaEl.innerHTML = ''` before deciding whether to show images or placeholder.
**Warning signs:** Multiple identical images appear in the media slot; test asserting single `<img>` finds multiple.

### Pitfall 4: Test 2d (slot mediów placeholder) regression
**What goes wrong:** The existing test at line 155-159 of `ElementInfoOverlay.test.js` asserts `media.textContent` contains `pl.modals.elementInfo.mediaPlaceholder` when `entry.media` is empty. If the refactored render path clears `mediaEl.textContent` without re-setting the placeholder, this test fails.
**Why it happens:** Phase 16 modifies the exact code branch the test covers.
**How to avoid:** The placeholder branch (`!entry?.media?.length`) must still set `mediaEl.textContent = pl.modals.elementInfo.mediaPlaceholder`. Only the non-empty branch adds `<img>` elements.
**Warning signs:** `ElementInfoOverlay — slot mediów placeholder (Test 2d)` fails.

### Pitfall 5: ATTRIBUTION.txt missing entry for tabliczka-znamionowa.webp (already exists)
**What goes wrong:** `public/media/` already contains `tabliczka-znamionowa.webp` from Phase 14. If ATTRIBUTION.txt is created listing only Phase 16 new files, the gate check finds a file with no entry → gate fails.
**Why it happens:** The gate requires "wpis dla KAZEGO pliku w public/media/" — covers pre-existing files too.
**How to avoid:** ATTRIBUTION.txt must include `tabliczka-znamionowa.webp` as first entry on creation.
**Warning signs:** Manual gate check: `ls public/media/` shows more files than ATTRIBUTION.txt has entries.

### Pitfall 6: Polish literals in src/media/MediaManager.js trigger boundary scanner
**What goes wrong:** If any Polish string literal (with diacritics) appears in a non-comment line of `MediaManager.js`, the `boundaries.test.js` Polish literal scanner fails.
**Why it happens:** `src/media/` is NOT in the `ALLOWED_PATHS` list (`['src/i18n/', 'src/training/scenarios/', 'src/data/']`). The scanner will scan `src/media/MediaManager.js`.
**How to avoid:** MediaManager has no user-facing strings. If any Polish text is needed, reference `pl.js`. In practice, MediaManager needs no Polish strings at all.

### Pitfall 7: onerror set after src assignment (timing edge case)
**What goes wrong:** In some environments (not jsdom, but Safari edge cases), setting `img.src` before `img.onerror` can cause the error to fire before the handler is attached.
**Why it happens:** Browser may synchronously resolve `src` for already-cached 404 responses.
**How to avoid:** Always attach `img.onerror` before assigning `img.src`. The Pattern 2 code above follows this order.

```javascript
// @vitest-environment node  (no DOM needed — pure service)
import { describe, it, expect, vi } from 'vitest';
import { MediaManager } from '../src/media/MediaManager.js';

describe('MediaManager — resolveSrc', () => {
  it('prefixes /media/ to filename', () => {
    const mm = new MediaManager();
    expect(mm.resolveSrc('photo.webp')).toBe('/media/photo.webp');

describe('MediaManager — validateSrc (injected fetchImpl)', () => {
  it('returns true when HEAD request ok', async () => {
    const fetchImpl = vi.fn().mockResolvedValue({ ok: true });
    const mm = new MediaManager({ fetchImpl });
    const result = await mm.validateSrc('/media/photo.webp');
    expect(result).toBe(true);
    expect(fetchImpl).toHaveBeenCalledWith('/media/photo.webp', { method: 'HEAD' });
  });

  it('returns false when HEAD request not ok (404)', async () => {
    const fetchImpl = vi.fn().mockResolvedValue({ ok: false });
    const mm = new MediaManager({ fetchImpl });
    expect(await mm.validateSrc('/media/missing.webp')).toBe(false);
  });

  it('returns false when fetch throws (offline)', async () => {
    const fetchImpl = vi.fn().mockRejectedValue(new Error('network'));
    const mm = new MediaManager({ fetchImpl });
    expect(await mm.validateSrc('/media/photo.webp')).toBe(false);
  });
});
```
